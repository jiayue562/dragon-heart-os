## 一、系统架构

### 1.1 架构总览

```
┌─────────────────────────────────────────────────┐
│                  龙心OS v7.0                      │
│  ┌─────────────────────────────────────────────┐ │
│  │  AI原生组织工作流 Skill                      │ │
│  │                                             │ │
│  │  输入：悟空的任务指令                         │ │
│  │  处理：任务分解器→执行引擎→标注器→统计器     │ │
│  │  输出：标注回复 + 瓶颈日志 + 压缩建议         │ │
│  └─────────────────────────────────────────────┘ │
│         ↓            ↓             ↓             │
│  ┌──────────┐ ┌──────────┐ ┌──────────────┐     │
│  │ 并行执行  │ │ 串行标记  │ │ 统计归档      │     │
│  │ 🟢 [AI]  │ │ 🔴 [人]  │ │ 📊 瓶颈率    │     │
│  └──────────┘ └──────────┘ └──────────────┘     │
└─────────────────────────────────────────────────┘
```

### 1.2 三层架构

| 层 | 组件 | 职责 |
|----|------|------|
| **感知层** | 任务分解器 | 分析悟空任务，识别[A]/[人]/[⇄]三类工作 |
| **执行层** | AI并行引擎 | 并行执行所有[A]任务，记录进度 |
| **记录层** | 统计归档引擎 | 记录瓶颈数据，输出分析报告 |

### 1.3 数据流

```
悟空指令
   ↓
[任务分解器] → 输出：任务分解树（标注版）
   ↓
[AI并行引擎] → 先执行所有🟢 [AI] 部分
   ↓              ↓              ↓
 完成A         完成B         完成C
   ↓
[收敛报告] → 标记🟡 [⇄] 等待处，呈现给悟空
   ↓
悟空介入 🔴 [人] 决策
   ↓
[AI并行引擎] → 继续执行后续工作
   ↓
[统计归档引擎] → 输出完成总结 + 瓶颈日志
```

## 二、数据模型

### 2.1 任务分解树（核心数据结构）

```typescript
interface TaskDecomposition {
  taskId: string;           // 自动生成的任务ID
  taskName: string;         // 任务名称
  complexity: 'L1' | 'L2' | 'L3'; // 复杂度层级
  createdAt: string;        // ISO 8601 时间戳
  
  steps: WorkStep[];        // 工作步骤序列
  bottlenecks: Bottleneck[]; // 串联瓶颈记录
  statistics: Statistics;    // 任务统计
}

interface WorkStep {
  id: string;               // 步骤ID
  category: 'AI' | 'HUMAN' | 'BOTTLENECK'; // 分类
  symbol: '🟢 [AI]' | '🔴 [人]' | '🟡 [⇄]'; // 标注符号
  description: string;      // 步骤描述
  status: 'pending' | 'in_progress' | 'completed' | 'blocked';
  dependencies: string[];   // 依赖的步骤ID列表
  result?: string;          // 完成结果摘要
}

interface Bottleneck {
  type: '信息综合' | '决策拍板' | '跨部门协调' | '上下文传递';
  severity: 1 | 2 | 3;     // 1=低, 2=中, 3=高
  description: string;
  suggestion?: string;      // 压缩建议
  compressed: boolean;      // 是否已被压缩
}

interface Statistics {
  aiSteps: number;          // 🟢 AI步数
  humanSteps: number;       // 🔴 人步数
  bottleneckCount: number;  // 🟡 瓶颈数
  totalSteps: number;       // 总步数
  bottleneckRate: number;   // 瓶颈率 = bottleneckCount / totalSteps
}
```

### 2.3 标注格式正式规范

为确保标注一致且可读，所有标注必须遵循以下格式：

```
## 🗺️ 任务分解（任务名称）
- 🟢 [AI] 执行内容 → 说明（可选）
- 🟡 [⇄] 等待xx → 说明（可选）
- 🔴 [人] xx决策 → 说明（可选）

## 📊 任务统计（可选，仅复杂任务）
| 指标 | 数值 |
|------|------|
| 🟢 AI并行步数 | N步 |
| 🔴 人串行步数 | N步 |
| 🟡 串联瓶颈数 | N处 |
| 瓶颈率 | XX% |
| 建议 | 具体建议 |
```

## 三、标注规则详情

### 3.1 判定矩阵

| 工作类型 | 🟢 [AI] | 🔴 [人] | 🟡 [⇄] |
|---------|---------|---------|---------|
| 信息搜索 | ✅ 完全可并行 | ❌ | — |
| 文档生成 | ✅ 有标准格式 | ⚠️ 需悟空定调 | ⚠️ 需先确认方向 |
| 代码编写 | ✅ 可自动生成 | ⚠️ 架构设计需人定 | ⚠️ 需悟空确认需求 |
| 知识学习 | ✅ 可自动分析 | ❌ | ⚠️ 学完需悟空检查质量 |
| 数据整理 | ✅ 完全自动化 | ❌ | — |
| 图片/图表生成 | ✅ 有明确指示 | ⚠️ 审美标准需人定 | ⚠️ 需悟空指定风格 |
| 决策/选择 | ❌ | ✅ 必须人定 | — |
| 质量审查 | ❌ | ✅ 受托楔子 | — |
| 定方向/战略 | ❌ | ✅ 悟空独有 | — |
| 跨环节等待 | — | — | ✅ 此处停 |

### 3.2 常见错误处理

| 错误场景 | 现象 | 正确处理 |
|---------|------|---------|
| 标注过多 | 简单任务也标注一堆 | L1任务不标注步，只在回复中标一句"此为纯AI工作" |
| 标注混淆 | 🟡 [⇄] 和 🔴 [人] 分不清 | 🟡 = 需要悟空介入后才能继续；🔴 = 必须悟空亲自做 |
| 漏标依赖 | AI步依赖决策但没标🟡 | 检查每个AI的前置条件是否都满足 |
| 统计不准 | 同一环节被重复计数 | 一个物理步骤只对应一个标注 |

### 3.3 特殊场景

**场景1：悟空说"继续"**
```
🟢 [AI] 读取对话历史和工作记忆
🟢 [AI] 恢复中断的任务上下文
🟡 [⇄] 确认上次中断的位置是否正确
🔴 [人] 确认是否需要调整方向
```

**场景2：悟空说"教我xxx"**
```
🟢 [AI] 准备教学材料（结构化内容）
🟡 [⇄] 确认悟空的基础掌握程度
🟢 [AI] 逐步讲解
🟡 [⇄] 确认悟空是否理解
🟢 [AI] 深入补充
🔴 [人] 悟空消化后确认是否满足需求
```

**场景3：悟空说"帮我诊断/分析"**
```
🟢 [AI] 读取相关上下文
🟢 [AI] 执行分析
🟡 [⇄] 呈现分析结果，等待悟空确认是否偏离
🔴 [人] 悟空提供领域直觉和判断
```

**场景4：悟空说"继续未完成的任务"**
```
🟢 [AI] 读取Memory + 今日日志
🟢 [AI] 恢复任务上下文
🟡 [⇄] 确认继续的方向
🔴 [人] 确认优先级
```

## 六、故障处理

### 6.1 标注不一致

**问题**：不同任务中标注粒度不一致
**解决**：以L2为标准粒度，L1简化，L3细化。标注前先判断复杂度。

### 6.2 悟空反馈"标注太多"或"标注不够"

**解决**：
- 标注太多：降级标注（L3→L2→L1）
- 标注不够：升级标注（L1→L2→L3）

### 6.3 瓶颈日志缺失

**解决**：每次完成时检查是否已记录。如未记录，提醒自己在当前回复中追加。

### 6.4 标注符号被误读

**解决**：规则明确规定——标注符号后面必须跟完整含义，不能只有符号。
